## 简介
本文档旨在为mcp-proxy项目提供全面的故障排除指南。文档涵盖了SSE连接中断、MCP插件启动失败、文档解析超时、OSS上传签名无效等典型场景的解决方案。通过分析全局错误处理机制、日志信息和信号处理，帮助开发者快速定位并解决系统运行中的问题。

## 常见问题及解决方案

### SSE连接中断
SSE（Server-Sent Events）连接中断可能由网络不稳定、服务器超时或客户端异常关闭引起。在`mcp-proxy`项目中，SSE服务器通过`run_sse_server`函数实现，该函数负责将stdio MCP服务器的输出通过SSE协议暴露给客户端。

当SSE连接中断时，首先检查网络连接状态，确保客户端与服务器之间的通信正常。其次，检查服务器端的日志，确认是否有"Process spawn failed"等错误信息。最后，验证客户端是否正确处理了SSE连接的生命周期，包括连接建立、数据接收和连接关闭。

**Section sources**

### MCP插件启动失败
MCP插件启动失败通常与命令执行环境、参数配置或依赖缺失有关。在`run_sse_server`函数中，通过`Command::new`创建子进程来启动MCP服务器，并正确处理了命令参数和环境变量的传递。

启动失败的常见原因包括：
- 命令路径错误或命令不存在
- 环境变量配置不正确
- 缺少必要的依赖库
- 权限不足

解决方案包括验证命令路径、检查环境变量配置、确保所有依赖已正确安装，以及以适当的权限运行程序。

**Section sources**
- [sse_server.rs](file://mcp-proxy/src/server/handlers/sse_server.rs#L0-L94)

### 文档解析超时
文档解析超时问题主要出现在`mineru_parser`组件中。该组件通过子进程调用MinerU工具进行文档解析，并设置了超时机制来防止长时间挂起。

在`mineru_parser.rs`文件中，解析超时时间通过以下逻辑确定：
```rust
let timeout_seconds = if self.config.timeout == 0 {
    3600
} else {
    self.config.timeout
};
```
默认超时时间为3600秒（1小时），如果配置为0则使用统一配置。当解析过程超过设定时间，系统会发送取消信号并返回"解析已取消"的错误。

解决解析超时的方法包括：
- 增加超时时间配置
- 优化待解析文档的大小和复杂度
- 检查MinerU环境配置是否正确
- 确保系统资源充足

### OSS上传签名无效
OSS上传签名无效问题通常与签名生成算法、时间戳或权限配置有关。在`private_client.rs`文件中，`PrivateOssClient`结构体负责生成OSS上传和下载的签名URL。

签名生成的关键代码如下：
```rust
fn generate_upload_url(
    &self,
    object_key: &str,
    expires_in: std::time::Duration,
    content_type: Option<&str>,
) -> Result<String> {
    let prefixed_key = self.config.get_prefixed_key(object_key);
    let mut builder = RequestBuilder::new().with_expire(expires_in.as_secs() as i64);
    if let Some(ct) = content_type {
        builder = builder.with_content_type(ct);
    } else {
        builder = builder.with_content_type("application/octet-stream");
    }
    let url = self.client.sign_upload_url(&prefixed_key, &builder);
    let url = utils::replace_oss_domain(&url);
    Ok(url)
}
```
签名无效的可能原因包括：
- AccessKey ID或AccessKey Secret配置错误
- 签名过期时间设置不当
- Bucket名称或Endpoint配置不正确
- 网络问题导致签名请求失败

解决方案包括验证OSS配置信息、检查网络连接、确保签名在有效期内使用，以及查看详细的错误日志。

## 全局错误处理机制

### 错误处理中间件
`document-parser`项目中的`error_handler.rs`文件实现了全局错误处理机制。该机制通过`global_error_handler`函数将不同类型的错误映射到相应的HTTP状态码和错误响应。

错误处理的核心逻辑如下：
```rust
pub async fn global_error_handler(err: AppError) -> impl IntoResponse {
    let (status, error_response) = match &err {
        AppError::Validation(_) => {
            warn!("Validation error: {}", err);
            (StatusCode::BAD_REQUEST, err.to_http_result::<()>())
        }
        AppError::Network(_) | AppError::Timeout(_) => {
            warn!("Network/Timeout error: {}", err);
            (StatusCode::REQUEST_TIMEOUT, err.to_http_result::<()>())
        }
        AppError::Parse(_) | AppError::MinerU(_) | AppError::MarkItDown(_) => {
            error!("Parser error: {}", err);
            (StatusCode::UNPROCESSABLE_ENTITY, err.to_http_result::<()>())
        }
        AppError::Database(_) | AppError::Internal(_) => {
            error!("Internal error: {}", err);
            (StatusCode::INTERNAL_SERVER_ERROR, err.to_http_result::<()>())
        }
        AppError::Oss(_) => {
            error!("OSS error: {}", err);
            (StatusCode::BAD_GATEWAY, err.to_http_result::<()>())
        }
        _ => {
            error!("Unknown error: {}", err);
            (StatusCode::INTERNAL_SERVER_ERROR, err.to_http_result::<()>())
        }
    };
    (status, Json(error_response))
}
```
该机制将错误分为不同级别，对应不同的HTTP状态码：
- 验证错误：400 Bad Request
- 网络/超时错误：408 Request Timeout
- 解析错误：422 Unprocessable Entity
- 数据库/内部错误：500 Internal Server Error
- OSS错误：502 Bad Gateway

## 日志分析技巧

### 关键错误信息识别
有效的日志分析是故障排除的关键。系统使用`tracing`库记录不同级别的日志信息，包括`info`、`warn`和`error`级别。

需要重点关注的错误信息包括：
- "Process spawn failed"：表示进程创建失败，通常与命令路径或权限有关
- "Storage upload error"：表示存储上传错误，可能与OSS配置或网络连接有关
- "MinerU stderr"：表示MinerU解析器输出了错误信息
- "Virtual environment creation failed"：表示虚拟环境创建失败

日志记录示例：
```rust
info!("HTTP {} {} - {} - {:?}", method, uri, response.status(), duration);
warn!("Validation error: {}", err);
error!("Parser error: {}", err);
```

## 信号处理与优雅关闭

### SIGTERM与SIGKILL差异
在Unix系统中，SIGTERM和SIGKILL是两种不同的进程终止信号，它们在优雅关闭（graceful shutdown）中有重要区别。

- SIGTERM（信号15）：请求进程终止。进程可以捕获此信号并执行清理操作，如保存状态、关闭连接等，然后再退出。这是实现优雅关闭的推荐方式。
- SIGKILL（信号9）：强制终止进程。进程无法捕获或忽略此信号，会立即终止，不执行任何清理操作。

### 优雅关闭实现
`voice-cli`项目中的`signal_handling.rs`文件提供了统一的信号处理工具，支持多种关闭信号的监听和处理。

核心实现如下：
```rust
pub async fn handle_system_signals() {
    let ctrl_c = async {
        if let Err(e) = signal::ctrl_c().await {
            error!("Failed to listen for Ctrl+C: {}", e);
        } else {
            info!("Received Ctrl+C signal");
        }
    };

    #[cfg(unix)]
    let terminate = async {
        use tokio::signal::unix::{signal, SignalKind};
        match signal(SignalKind::terminate()) {
            Ok(mut stream) => {
                stream.recv().await;
                info!("Received SIGTERM signal");
            }
            Err(e) => {
                error!("Failed to listen for SIGTERM: {}", e);
            }
        }
    };

    #[cfg(not(unix))]
    let terminate = std::future::pending::<()>();

    tokio::select! {
        _ = ctrl_c => debug!("Ctrl+C signal handled"),
        _ = terminate => debug!("SIGTERM signal handled"),
    }
}
```
该实现同时监听Ctrl+C（SIGINT）和SIGTERM信号，允许服务在收到终止信号时执行清理操作，实现优雅关闭。

## 环境相关问题排查

### 网络配置
网络配置问题是导致服务不可用的常见原因。排查步骤包括：
1. 检查网络连接是否正常
2. 验证防火墙设置，确保必要的端口已开放
3. 检查DNS解析是否正常
4. 验证代理设置（如果使用代理）

对于OSS连接问题，特别需要检查：
- Endpoint配置是否正确
- 网络是否能够访问OSS服务
- 是否需要配置代理

### 权限设置
权限问题可能导致文件读写失败、目录访问拒绝等错误。排查步骤包括：
1. 检查文件和目录的读写权限
2. 验证进程运行用户是否有足够的权限
3. 检查特殊权限设置（如SELinux、AppArmor）

系统为不同操作系统提供了具体的权限修复建议：
- Unix系统：使用`chmod`和`chown`命令修改权限和所有者
- Windows系统：以管理员身份运行或调整UAC设置

### 依赖缺失
依赖缺失可能导致模块无法加载或功能异常。排查步骤包括：
1. 检查Cargo.toml中的依赖声明
2. 运行`cargo check`验证依赖完整性
3. 检查本地环境是否安装了必要的系统库
4. 验证Python虚拟环境（如果使用）是否正确创建

### 使用基准测试
`benches`目录下的基准测试可以用于复现性能相关问题。例如：
- `document_parsing_bench.rs`：文档解析性能测试

通过运行这些基准测试，可以：

3. 识别性能瓶颈

`tests`目录下的集成测试可以用于复现功能相关问题。例如：
- `integration_tests.rs`：OSS客户端集成测试
- `image_processing_integration_tests.rs`：图片处理集成测试

集成测试的优势包括：
1. 在接近生产环境的条件下测试功能
2. 验证多个组件之间的交互
3. 捕获边界条件和异常情况

